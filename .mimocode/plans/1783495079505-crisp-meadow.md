## Phase 1：性能瓶颈修复 + 架构解耦（立即执行）

### 1.1 [P1-01] 录音同步 I/O 迁移为异步

**现状**: `media.rs:629-639` 在 `std::sync::Mutex` 内调用 `std::fs::File::write_all()` / `seek()` / `flush()`，每个 RTP 包触发多次同步 syscall，阻塞 tokio worker 线程。

**问题**:
| # | 严重级别 | 问题描述 | 涉及文件 |
|---|---------|---------|---------|
| 1 | 🔴 高 | 录音 sync I/O 阻塞 tokio runtime，高并发下会导致整个进程响应退化 | `media.rs:629-639, 727-844` |
| 2 | 🔴 高 | `std::sync::Mutex` 在 async 上下文中使用，违反 AGENTS.md 5.4 异步编程规范 | `media.rs:635` |

**重构方案**:
1. 将 `WavCallRecorder` 改为使用 `tokio::fs::File` + `tokio::task::spawn_blocking` 封装
2. 录音路径改为：RTP 包 → channel 发送到独立录音 task → 独立 task 做文件 I/O
3. 去掉 `Mutex`，改为 `mpsc` channel 每个录音 session 独立一个 task

**关键代码思路**:
```rust
// 重构前 (media.rs:629-639)
fn record_rtp_packet(&self, relay_port: u16, packet: &RtpPacket) -> Result<bool, MediaError> {
    let recording = self.recordings.get(&relay_port).map(|v| v.clone());
    let Some(recording) = recording else { return Ok(false); };
    let mut recorder = recording.recorder.lock().expect("lock poisoned");
    recorder.record(recording.channel, packet)  // sync I/O inside Mutex!
}

// 重构后：channel-based async recording
struct AsyncRecorder {
    tx: mpsc::UnboundedSender<(RecordingChannel, Vec<u8>)>,
}

impl AsyncRecorder {
    fn record(&self, channel: RecordingChannel, packet: &RtpPacket) {
        let samples = decode_to_pcm16(channel, packet);  // 解码到 PCM
        let _ = self.tx.send((channel, samples));  // 非阻塞发送
    }
}

// 独立 tokio task 处理文件 I/O
tokio::spawn(async move {
    let mut file = tokio::fs::File::create(path).await?;
    while let Some((channel, samples)) = rx.recv().await {
        file.write_all(&samples).await?;  // 不阻塞其他 task
        // 定期 flush
    }
});
```

**涉及文件**: `media.rs` (重构 `WavCallRecorder` + `record_rtp_packet` + 新增 `async_recorder` 模块)

### 1.3 [P1-03] SBC RateLimiter 分片化

**现状**: `sbc.rs:86-118`，`RateLimiter` 使用单个 `Mutex<HashMap<IpAddr, TokenBucket>>`。每个入站 SIP 包都要加锁检查。

**问题**:
| # | 严重级别 | 问题描述 | 涉及文件 |
|---|---------|---------|---------|
| 1 | 🔴 高 | 高 CPS 下单 Mutex 成为全局瓶颈，所有 SIP 收包串行化 | `sbc.rs:88, 106` |
| 2 | 🟡 中 | stale entry eviction 在锁内执行，增加持锁时间 | `sbc.rs:109-111` |

**重构方案**:
```rust
// 重构后：DashMap 分片
pub struct RateLimiter {
    buckets: DashMap<IpAddr, TokenBucket>,
    capacity: f64,
    fill_rate: f64,
}

impl RateLimiter {
    pub fn check_rate(&self, ip: IpAddr) -> bool {
        let now = Instant::now();
        let mut bucket = self.buckets.entry(ip)
            .or_insert_with(|| TokenBucket::new(self.capacity));
        bucket.take(self.capacity, self.fill_rate, now)
    }
}
```

**涉及文件**: `services/sip-edge/src/sbc.rs`

### 1.4 [P1-04] RTP 收发热路径 DashMap 访问合并

**现状**: 每个 RTP 包在 `relay_media_port()` (media.rs:1377-1478) 中触发 6-8 次独立 DashMap shard lock 获取 (`record_metric` x3, `target_for_port`, `learn_symmetric_source`, `record_rtp_packet`, `process_dtmf_packet`)。

**问题**:
| # | 严重级别 | 问题描述 | 涉及文件 |
|---|---------|---------|---------|
| 1 | 🟡 中 | 单包 6-8 次 DashMap 锁获取，高 pps 下 cache line bouncing | `media.rs:1403-1476` |
| 2 | 🟡 中 | `record_metric` 每次调用获取 DashMap entry 再回调，开销叠加 | `media.rs:498-523` |

**重构方案**: 将每个 port 的所有状态合并为单个 `Arc<RtpPortState>`，使用原子计数器替代 DashMap per-port 结构：
```rust
struct RtpPortState {
    target: AtomicU64,           // compact SocketAddr as u64
    peer_port: AtomicU16,
    // 统计计数器全部用 AtomicU64，无锁
    received: AtomicU64,
    forwarded: AtomicU64,
    dropped_invalid: AtomicU64,
    dropped_no_target: AtomicU64,
    send_errors: AtomicU64,
}
```
一个 `DashMap<u16, Arc<RtpPortState>>` 替代原来 6 个 DashMap。

**涉及文件**: `services/sip-edge/src/media.rs`

### 1.5 [P1-05] RTP 收发 Buffer Pool

**现状**: `media.rs:1385` 每个 port 分配一个 65KB `Vec<u8>` 作为接收 buffer。RTP 解析 (`rtp-core/src/packet.rs:43-118`) 每包分配 `Vec<u8>` payload。

**问题**:
| # | 严重级别 | 问题描述 | 涉及文件 |
|---|---------|---------|---------|
| 1 | 🟡 中 | 无 buffer pool，RTP 解析每包 clone payload Vec | `rtp-core/src/packet.rs:85, 115` |

**重构方案**: 引入 `bytes::Bytes` + `bytes::BytesMut` 池化 buffer，RTP 解析直接借用 `&[u8]` 返回借用类型，不 clone payload。

**涉及文件**: `crates/rtp-core/src/packet.rs`, `services/sip-edge/src/media.rs`

---

### 1.6 [P1-06] SIP Parser 引入零拷贝借用

**现状**: `sip-core/src/message.rs:62` 使用 `String::from_utf8_lossy(head)` 拷贝整个 header 块。`header.rs:33-37` 每个 header 值都 `.to_string()`。`uri.rs:6-12` 每个 URI 字段都 heap 分配。

**问题**:
| # | 严重级别 | 问题描述 | 涉及文件 |
|---|---------|---------|---------|
| 1 | 🟡 中 | SIP 解析每消息 10+ 次 heap alloc，不符合零拷贝目标 | `sip-core/src/message.rs:62` |
| 2 | 🟢 低 | HeaderMap 用 Vec 线性扫描 O(n)，小 n 可接受 | `sip-core/src/header.rs:53` |

**重构方案**: SIP 解析改为基于 `&'a [u8]` / `&'a str` 借用类型，header 值使用 `Cow<'a, str>`。需要在 crate 接口上引入生命周期参数。

**涉及文件**: `crates/sip-core/src/message.rs`, `header.rs`, `uri.rs`

## Phase 2：功能补全 + 性能提升（1-3 个月）

### 2.1 [P2-01] 路由引擎 Trie 化

**现状**: `call-core/src/routing.rs:207-209` 使用 `Vec<Route>` 线性扫描 `starts_with()`，每次路由查询遍历全部路由。

**重构方案**: 引入前缀 Trie 数据结构，路由加载时编译为不可变 Trie，查询时间从 O(n) 降至 O(k)（k=号码长度）。

**涉及文件**: `crates/call-core/src/routing.rs` (新增 `prefix_trie.rs`)

---

### 2.2 [P2-02] 实时余额扣减

**现状**: 余额检查在呼叫建立前查一次 DB，通话中无实时扣减，挂断后批量 reconcile。

**重构方案**: 引入内存余额缓存 `DashMap<AccountId, AtomicI64>` + CAS 操作，通话开始时预扣最小费用，通话中按秒累加扣减，挂断时 finalize。

**涉及文件**: `crates/cdr-core/src/lib.rs` (新增 `balance_cache.rs`), `services/sip-edge/src/` (计费集成)

---

### 2.3 [P2-03] RTP 转码从主路径卸载

**现状**: `transcode.rs` 存在但未接入 RTP relay。录音中有内联 G.711 解码。

**重构方案**: 转码放入独立 `tokio::task::spawn_blocking` 池执行，不阻塞 RTP 收发 loop。

---

### 2.4 [P2-04] Dialog 完整状态机

**现状**: `dialog.rs` 仅做 validation（85 行），无 RFC 3261 Section 12 Dialog 状态机（Init/Early/Confirmed/Terminated）。

**重构方案**: 在 `call-core` 中新增完整 Dialog FSM，管理 Route Set、CSeq、SDP 状态。

## 4. 架构设计

### 4.1 分层架构（进程内模块划分）

vos-server 单进程内包含：
- SIP 信令层 (B2BUA, UDP/TCP/TLS/WS)
- 路由引擎 (Prefix Match / LCR / Failover)
- 计费引擎 (Realtime Balance / CDR Writer / Rating)
- 媒体控制器 (RTP Relay / Recording / Transcode / DTMF)
- 安全模块 (SBC / ACL / Auth / Anti-Fraud)
- API 层 (Axum REST)
- 共享状态层 (DashMap + NATS + sqlx Pool)

### 4.2 信令与媒体分离原则

信令节点（轻量级，CPU 密集）与媒体节点（重量级，I/O + CPU 密集）应可独立扩展。

### 4.3 核心设计模式

- 依赖注入：通过构造函数注入依赖，方便测试
- Result 类型错误处理：所有可恢复错误返回 Result，不使用 panic
- Builder 模式：复杂对象使用 Builder 构建
- Repository 模式：数据访问抽象化（cdr-core 提供 trait）

### 4.4 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 数据库 ORM | sqlx（编译期检查） | 类型安全、性能好 |
| 错误处理 | thiserror (库) + anyhow (应用) | 分层错误处理 |
| 序列化 | serde + serde_json | Rust 生态标准 |
| 日志 | tracing + tracing-subscriber | 结构化、异步友好 |
| 并发数据结构 | DashMap (分片锁) | 高并发读多写少场景 |
| 消息队列 | NATS JetStream | CDR 事件流，批量写入 |
| SIP 解析 | 自研 (sip-core, 零外部依赖) | 轻量、可控 |

### 4.5 并发状态管理

| 层级 | 数据结构 | 锁类型 |
|------|---------|--------|
| SIP Dialogs | DashMap<CallId, Arc<InboundTransaction>> | DashMap 分段锁 |
| RTP Sessions | DashMap<u16, SocketAddr> 等 6 个 | DashMap 分段锁 |
| Media Inner | Arc<Mutex<MediaRelayStateInner>> | std::sync::Mutex |
| 路由表 | Arc<RwLock<RouteTable>> | std::sync::RwLock |
| CDR 缓存 | Mutex<Vec<CallCdr>> | std::sync::Mutex |
| SBC RateLimiter | Mutex<HashMap<IpAddr, TokenBucket>> | std::sync::Mutex |
